# 时间类型与自学习设计

## 一、决策链路总览

```
用户输入 → AI 判断任务类型 (smartCategory)
                ↓
         脚本根据任务类型 + 用户配置 → 选择时间槽 (slot)
                ↓
         时间槽 + 总时间补全表 → 生成最终提醒时间
```

**核心原则**：时间补全由「AI 判断任务类型」+「用户配置的时间类型」决定。自学习数据仅作为 AI 上下文参考，不直接控制时间槽。

## 二、五种时间类型定义

> **[2026-06-17 修改]** 五种时间类型调度逻辑改为「开始时间 + 偏移量」模式，不再依赖时间补全表。

### 1. 工作时间 (`work`)

- **适用日期**：仅 Off day 判定的工作日
- **适用时段**：`workStart`（默认 08:30）至 `workEnd`（默认 17:30）
- **调度规则**：
  - 如果当前时间 < `workStart`，安排在当天 `workStart`
  - 如果当前时间 ≥ `workStart`，安排在当前时间 + `floatingTaskDelayMinutes`（默认 10 分钟）
  - 如果候选时间超出 `workEnd`，顺延到下一个工作日的 `workStart`
- ~~补全规则**：在工作时间范围内，按总时间补全表寻找下一个可用补全时间~~ ← 2026-06-17 废弃
- ~~跨天逻辑**：如果当天工作时间已结束，跳到下一个 Off day 判定的工作日，从 `workStart` 开始补全~~ ← 2026-06-17 废弃
- ~~边界判断**：候选时间 + 任务预计时长不能超过 `workEnd`；超过则顺延到下一个工作日~~ ← 2026-06-17 废弃

### 2. 休息时间 (`restTime`)

- **适用日期**：所有日期（工作日 + 休息日）
- **适用时段**：
  - 工作日：`dailyTaskStart`（默认 20:00）至 `quietAfter`（默认 22:00）
  - 休息日：全天
- **调度规则**：
  - **工作日**：
    - 如果当前时间 < `dailyTaskStart`，安排在当天 `dailyTaskStart`
    - 如果当前时间 ≥ `dailyTaskStart`，安排在当前时间 + `floatingTaskDelayMinutes`（默认 10 分钟）
    - 如果候选时间超出 `quietAfter`，顺延到下一天
  - **休息日**：
    - 按照休息日的补全规则来（复用 `restDay` 的调度规则）
    - 如果当前时间 < `weekendDefaultTime`（默认 10:00），安排在当天 `weekendDefaultTime`
    - 如果当前时间 ≥ `weekendDefaultTime`，安排在当前时间 + `floatingTaskDelayMinutes`（默认 10 分钟）
    - 全天可用，无边界限制
- **补全规则**：
  - 工作日：在 `dailyTaskStart` ~ `quietAfter` 范围内，按开始时间 + 偏移量调度
  - 休息日：按照休息日的补全规则（复用 `restDay` 的调度规则）

### 3. 休息日 (`restDay`)

- **适用日期**：仅 Off day 判定的休息日
- **适用时段**：全天
- **调度规则**：
  - 如果当前时间 < `weekendDefaultTime`（默认 10:00），安排在当天 `weekendDefaultTime`
  - 如果当前时间 ≥ `weekendDefaultTime`，安排在当前时间 + `floatingTaskDelayMinutes`（默认 10 分钟）
  - 如果今天不是休息日，顺延到下一个休息日的 `weekendDefaultTime`
- ~~补全规则**：在休息日全天，按总时间补全表寻找下一个可用补全时间~~ ← 2026-06-17 废弃

### 4. 指定时间 (`specified`)

- **适用规则**：按用户为该任务类型指定的具体时间补全（**不受本次修改影响**）
- **日期约束**（`specifiedDayConstraint`）：
  - `any`：不限日期，任何日子都可安排
  - `work`：仅在 Off day 判定的工作日安排
  - `rest`：仅在 Off day 判定的休息日安排
  - `split`：工作日和休息日分别使用不同的指定时间列表（`workDayTimes` / `restDayTimes`）
- **跨天逻辑**：如果今天所有指定时间已过，自动跳到下一个符合日期约束的日期
- **多个指定时间**：取所有指定时间中最近可用的一个

### 5. 任何时间 (`any`)

- **适用日期**：不限
- **适用时段**：不限
- **调度规则**：
  - 使用 `generalPreferredTime`（默认 20:00）作为开始时间
  - 如果当前时间 < `generalPreferredTime`，安排在当天 `generalPreferredTime`
  - 如果当前时间 ≥ `generalPreferredTime`，安排在当前时间 + `floatingTaskDelayMinutes`（默认 10 分钟）
- ~~补全规则**：直接按总时间补全表寻找下一个可用补全时间，不做日期/时段过滤~~ ← 2026-06-17 废弃

## 三、任务类型到时间类型的映射

### 3.1 用户配置优先

用户在「简单任务偏好」中为每种任务类型设置了安排方式（mode），对应关系：

| 用户设置 mode | 实际时间槽 |
|---|---|
| `work` | 工作时间 |
| `restTime` / `evening` | 休息时间 |
| `restDay` / `weekend` | 休息日 |
| `specified` | 指定时间 |
| `any` | 任何时间 |

### 3.2 无配置时的默认映射

| 任务类型 (TaskCategory) | 默认时间槽 |
|---|---|
| `work` | 工作时间 |
| `shopping` | 取决于 `shoppingMode`，默认任何时间 |
| `housework` | 取决于 `houseworkMode`，默认任何时间 |
| `study` | 取决于 `studyMode`，默认指定时间 |
| `health` | 取决于 `exerciseMode`，默认指定时间 |
| `relationship` | 休息时间 |
| `finance` | 休息时间 |
| `leisure` | 休息日 |
| `general` / 其他 | 任何时间 |

### 3.3 自定义任务类型

用户在设置中创建的自定义任务（`customSimpleTasks`）和内置任务（`simpleTaskPreferences`）具有最高优先级。如果匹配到自定义任务的 `typeId`，直接使用其 `mode` 转换为时间槽。

## 五、自学习机制（当前状态：仅积累）

### 5.1 学习数据来源

脚本在用户**显式设置了时间**的任务上记录学习数据（`learnFromExplicitSchedule`）：

- **按任务类型学习**：写入 `config.userPreferences.learning[category]`，如 `learning.work.restTime += 1`
- **按提醒列表学习**：写入 `config.userPreferences.learningByList[listName]`，如 `learningByList.Reminders.evening += 1`

### 5.2 学习数据结构

```json
{
  "learning": {
    "general": { "morning": 1, "work": 0, "restTime": 5, "restDay": 1, "total": 7 },
    "shopping": { "restTime": 1, "restDay": 1, "total": 2 }
  },
  "learningByList": {
    "Reminders": { "evening": 4, "rest": 1, "restTime": 2, "restDay": 2, "total": 10 }
  }
}
```

字段映射：
- `morning` → 上午
- `work` → 工作时间
- `restTime` / `evening` / `rest` → 休息时间（合并计算）
- `restDay` / `weekend` → 休息日（合并计算）
- `any` → 任何时间

### 5.3 当前状态：暂停控制

**代码位置**：`src/smartScheduler.ts` → `decideSmartScheduleDecisionAsync()`

```ts
// 暂停：自学习样本仍在积累阶段，暂不让 learning / learningByList 控制时间槽。
// 后续样本足够稳定后，可再恢复：learnedTaskSlot(category, config) || learnedListSlot(reminderListCategory, config)
const learned: TimeSlot | undefined = undefined
const slot = defaultSlot(category, config)
```

- 学习数据**仍在持续记录**（`learnFromExplicitSchedule` 不受影响）
- 但决策时**不再使用**学习数据覆盖用户配置的时间类型
- 原因：当前样本量太少（多数类型 total < 10），学习结果偏差大

### 5.4 恢复计划

当学习数据积累到足够稳定后（建议至少 1 个月），可以恢复学习控制：

1. 将 `decideSmartScheduleDecisionAsync()` 中的 `learned` 改回：
   ```ts
   const learned = learnedTaskSlot(category, config) || learnedListSlot(reminderListCategory, config)
   const slot = learned || defaultSlot(category, config)
   ```
2. 恢复后，学习结果仅作为**默认槽之后的覆盖**，用户明确配置的简单任务偏好（`configuredSimpleTaskFromCategory`）仍应优先
3. 建议增加最小样本阈值（当前是 `total >= 3` 且 `best.count >= 2`），可适当提高到 `total >= 10`

## 六、Off day 依赖说明

时间类型的日期判断（工作日 / 休息日）依赖 Off day 脚本：

- **调用方式**：`import { getRestDayInfo } from "../../Off day/utils/is_rest_day"`
- **判断优先级**：
  1. 手动设置的休息/工作日
  2. 已同步的节假日日历（含法定放假和调休上班）
  3. 固定每周休息日（默认周六、周日）
- **异步特性**：`getRestDayInfo()` 是 async 函数，Alertpilot 已改为 `await` 调用
- **当前 Off day 能力**：
  - ✅ `getRestDayInfo(dateKey)` — 获取指定日期详情
  - ✅ `isRestDay(dateKey)` — 判断是否休息日
  - ✅ `getLatestRestDayInfo()` — 从今天开始找最近休息日
  - ❌ **缺少** `getLatestWorkDayInfo()` — 从今天开始找最近工作日（可后续补充）

## 七、冲突重排机制

智能调度确定时间后，`avoidReminderTimeConflicts` 会检查新任务是否与已有提醒冲突。

### 7.1 范围重叠检测

`overlaps` 使用 `rangesOverlap` 判断新任务的时间范围是否与其他任务的时间范围有交集。当重叠数达到 `maxConcurrentTasks`（默认 2）上限时，新任务会被顺延。

### 7.2 顺延策略

顺延时间 = 重叠任务中最晚的结束时间 + `conflictBufferMinutes`（默认 10 分钟）。

**示例**（`generalTaskDurationMinutes` = 40 分钟，`conflictBufferMinutes` = 10 分钟）：

```
09:30 任务:    [09:30 ======== 10:10]   ← 持续 40 分钟，10:10 结束
10:00 任务:              [10:00 ======== 10:40]
新任务:                  [10:00 ======== 10:40]
                              ↑ 09:30 任务还在这里（10:10 结束）

→ overlaps = 2（09:30 任务 + 10:00 任务）→ 达到上限
→ latestEnd = max(10:10, 10:40) = 10:40
→ 新任务开始时间 = 10:40 + 10 分钟缓冲 = 10:50
```

### 7.3 相关配置

| 变量 | 默认值 | 说明 |
|---|---|---|
| `maxConcurrentTasks` | 2 | 时间范围重叠的任务最多允许数量 |
| `conflictBufferMinutes` | 10 | 从重叠任务最晚结束时间起的缓冲分钟数 |
| `conflictDeferMinutes` | 30 | 无指定时间任务的顺延步长（用于时间补全） |
| `generalTaskDurationMinutes` | 30 | 一般任务默认耗时（估算重叠范围） |

### 7.4 紧急任务调度

紧急任务（带 `!` 标记或设置了 iOS 优先级）的调度行为与普通任务不同：

1. **保留原时间**：紧急任务始终保留在用户指定的时间，不会被顺延
2. **查询范围缩小**：只检查开始时间前后 `URGENT_CONFLICT_LOOKAROUND_MINUTES`（默认 60 分钟）内的冲突
3. **顺延已有任务**：`makeRoomForUrgentTask` 会把冲突的低优先级已有任务顺延，为紧急任务腾出空间
4. **最多顺延 5 个**已有任务（`MAX_URGENT_MOVED_REMINDERS`）

```
普通任务：新任务 → 找空位 → 可能被顺延
紧急任务：新任务 → 保留原时间 → 冲突的旧任务被顺延
```

紧急任务的优先级判断：
- 文本中包含 `!`、`!!`、`!!!`（或中文 `！`）标记
- iOS 提醒优先级设置为高（值 1）/ 中（值 2/5）/ 低（值 3/9）
